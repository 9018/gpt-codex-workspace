P0：Supervisor Loop 的 start/stop 测试当前失败
我实际运行了：
backend/test/supervisor-review/*.test.mjs
backend/test/execution-core/checkpoint-supervisor-loop.test.mjs
大部分测试通过，但出现一个失败：
✖ checkpoint-supervisor-loop start/stop works

Promise resolution is still pending
but the event loop has already resolved
问题来自当前 stop()：
function stop() {
  _stopped = true;
  _running = false;

  if (_timer) {
    clearTimeout(_timer);
    _timer = null;
  }
}
start() 返回的 Promise 只会在下一次 poll() 进入时调用 resolve()。
但 stop() 把 timer 清掉之后，下一次 poll 永远不会发生，因此 Promise 不会 resolve。
应该持久保存 resolver：
let _resolveStart = null;

async function start(runId, options = {}) {
  if (_running) return;

  _running = true;
  _stopped = false;

  return new Promise((resolve) => {
    _resolveStart = resolve;

    const poll = async () => {
      if (_stopped || reachedMaxLoops()) {
        finish();
        return;
      }

      await safeTick(runId, options);

      if (_stopped) {
        finish();
        return;
      }

      _timer = setTimeout(poll, pollIntervalMs);
    };

    poll();
  });
}

function finish() {
  _running = false;

  if (_timer) {
    clearTimeout(_timer);
    _timer = null;
  }

  const resolve = _resolveStart;
  _resolveStart = null;
  resolve?.();
}

function stop() {
  _stopped = true;
  finish();
}
这个问题不是小测试瑕疵，而是可能造成：
* shutdown 卡住
* restart 等待不结束
* 测试进程悬空
* supervisor runtime 无法优雅停止

P1：
planStore.readPlan(null)
明显不正确
当前 Command Executor：
deps.planStore.readPlan(null).catch(() => ({}))
这是一个危险实现。
正确 plan ID 应来自：
run.supervisor_plan_id
应该先读取 Run，再读取 Plan：
const run =
  await deps.runStore.readRun(command.run_id);

const [currentRevision, lease, plan] =
  await Promise.all([
    deps.revisionReader.current(command.run_id),
    deps.leaseStore.read(command.run_id),
    run.supervisor_plan_id
      ? deps.planStore.readPlan(run.supervisor_plan_id)

  ]);
当前写法会导致：
* autonomy budget 不生效
* checkpoint policy 缺失
* takeover budget 缺失
* Action Guard 使用空 plan
* 错误被 .catch(() => ({})) 静默吞掉
这与方案中的 fail-closed 原则不一致。

五、最终判断
这次修改不是偏离方案，总体上确实是按照方案推进的，而且模块拆分、持久化、幂等和测试都比之前成熟很多。
但当前停留在：
核心零件基本造完
还没有到：
整车可以自动运行
最关键的剩余链路是：
ChatGPT 定时任务
    ↓
实际可见的 MCP review tool
    ↓
获得完整 Review Packet
    ↓
提交 Decision
    ↓
Command Store
    ↓
生产 Worker 自动 claim
    ↓
Command Executor
    ↓
send correction / resume / takeover / handoff
目前缺失或未确认接通的是：
工具注册
Worker 启动
Resume command route
Handoff command route
单一 checkpoint 主权
Supervisor loop 生命周期修复
所以准确结论是：
方案主体已经实现，但运行闭环还没有完成；当前代码属于“架构实现后期、生产接线前期”。
